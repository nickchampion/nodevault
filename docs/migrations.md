# Database Migrations

Migrations are managed by [drizzle-kit](https://orm.drizzle.team/docs/kit-overview). The TypeScript schema in `components/nodevault/domain/models/` is the single source of truth — migration SQL is **generated by diffing the schema**, not written by hand (except custom migrations, below).

Layout:

```
components/nodevault/domain/models/     Drizzle schema, one file per domain area (account.ts, auth.ts, ...)
components/postgres/migrations/       Generated, numbered SQL files (0000_..., 0001_...) + meta/ journal
drizzle.config.ts             drizzle-kit config (schema glob, output dir, connection)
```

Scripts (package.json):

| Script | Command | Purpose |
|---|---|---|
| `pnpm run db:generate` | `drizzle-kit generate` | Diff schema against migration history, emit a new SQL file |
| `pnpm run db` | `drizzle-kit migrate` | Apply pending migrations to the database |
| `pnpm run db:studio` | `drizzle-kit studio` | Browse the database in a UI |

## Everyday flow: schema change

1. **Edit the schema** in `components/nodevault/domain/models/`:
   - Changing an existing table: edit its file directly.
   - New table in an existing domain area: add the `pgTable` + inferred type (`export type X = typeof x.$inferSelect`) to that file.
   - New domain area: create `components/nodevault/domain/models/<area>.ts` and add `export * from './<area>.js'` to `components/nodevault/domain/models/index.ts` — that barrel is what feeds Drizzle's `schema` namespace, so `session.db.query.<table>` typing follows automatically.

2. **Generate the migration** (name it — the default names are random words):

   ```bash
   pnpm run db:generate --name=add-invoices
   # -> components/postgres/migrations/0001_add-invoices.sql
   ```

   (pnpm passes flags straight through to drizzle-kit — no `--` separator.)

3. **Review the generated SQL.** drizzle-kit is good but not psychic — renames in particular can come out as drop+create (it will prompt when ambiguous). Check destructive statements before applying.

4. **Apply it:**

   ```bash
   pnpm run db
   ```

5. **Commit the schema change, the SQL file, and the `meta/` journal together.** They are one unit; the journal is how drizzle-kit knows what has been generated.

## Custom migrations (data fixes, SQL drizzle-kit can't infer)

For backfills, data corrections, extensions, triggers, or anything else that isn't a schema diff:

```bash
pnpm run db:generate --custom --name=backfill-account-names
```

This emits an **empty** numbered SQL file in `components/postgres/migrations/` — write your SQL in it, then apply with `pnpm run db`. It runs in sequence with the schema migrations, and each migration is applied inside a transaction. Use `--> statement-breakpoint` between statements (see the existing files).

## Running migrations

`pnpm run db` applies every not-yet-applied migration, in order. Applied state is tracked in the target database itself (the `drizzle.__drizzle_migrations` table), so running it repeatedly is safe — already-applied files are skipped.

Connection resolution (see `drizzle.config.ts`): `DATABASE_URL` if set, otherwise the local dev default `postgres://nodevault:nodevault@localhost:5432/nodevault`.

```bash
# local dev (default URL, dev container must be up)
docker start nodevault-postgres   # pgvector/pgvector:pg18, once per boot
pnpm run db

# any other environment
DATABASE_URL=postgres://user:pass@host:5432/db pnpm run db
```

## Rules

- **Never edit a migration that has been applied anywhere** (including a teammate's machine or prod). Fix forward with a new migration.
- Don't hand-edit `meta/` — it's drizzle-kit's journal.
- Don't create SQL files in the migrations folder by hand; always go through `db:generate` (with `--custom` for hand-written SQL) so the journal stays consistent.
- Migrations run in filename order — the numeric prefix is assigned by drizzle-kit; don't renumber.
- To remove a generated-but-**never-applied** migration, run `pnpm exec drizzle-kit drop` (interactive) — it deletes the SQL file and fixes the `meta/` journal for you.

## Troubleshooting

- **"No schema changes, nothing to migrate 😴"** from `db:generate`: the schema and migration history already match — there's nothing to diff. If you expected a change, check the model file is exported from `components/nodevault/domain/models/index.ts` (drizzle-kit reads the glob, but the app's runtime schema comes from the barrel — keep them in sync).
- **Reset the local dev database** (throwaway data only):

  ```bash
  docker rm -f nodevault-postgres
  docker run -d --name nodevault-postgres -e POSTGRES_USER=nodevault -e POSTGRES_PASSWORD=nodevault \
    -e POSTGRES_DB=nodevault -p 5432:5432 pgvector/pgvector:pg18
  pnpm run db
  ```
